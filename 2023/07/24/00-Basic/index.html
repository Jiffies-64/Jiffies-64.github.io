原理篇

volatile原理

volatile的底层实现原理是内存屏障，Menory Barrier（Memory Fence）

  • 对volatile变量写指令后会加入写屏障
  • 对volatile变量读指令前会加入读屏障

1. 内存屏障

Menory Barrier（Memory Fence）

  • 可见性

    • 写屏障（sfence）保证在该屏障之前的，对共享变量的改动，都同步到主存当中
    • 读屏障（ifence）保证在该屏障之后的，对共享变量的读取，加载的是主存中的最新数据

  • 有序性

    JVM会在不影响正确性的前提下调整语句的执行顺序，在多线程环境下可能会有问题

    • 写屏障确保指令重排序时，不会将写屏障之前的代码排在写屏障之后
    • 读屏障保证指令重排序时，不会将读屏障之后的代码排在读屏障之前

1.1. 举个例子

情况1：线程1先执行，进入false分支

情况2：线程2先执行，执行到num=2，切换到线程1，还是进入else分支

情况3：线程2先执行，执行完毕，切换到线程1，进入if分支

但如果对线程2两条指令进行了重排序，对于情况2先执行第一条，num此时尚且为0，最终结果r.r1为0

2. 如何保证可见性

  • 写屏障保证在该屏障之前的，对共享变量的改动，都同步到主存当中

  • 读屏障保证在该屏障之后，对共享变量的读取，加载的是主存中最新的数据

3. 如何保证有序性

  • 写屏障确保指令重排序时，不会将写屏障之前的代码排在写屏障之后
  • 读屏障保证指令重排序时，不会将读屏障之后的代码排在读屏障之前

4. 局限

  • 如写屏障只能保证写屏障之前的代码不会排在写屏障之后，但不能保证写屏障之后的读操作排在写屏障之前
  • 有序性的保证也仅仅是本线程的相关代码不被重排序

2. Monitor

Monitor被翻译为监视器或管程

每个Java对象都可以关联一个Monitor对象，如果使用sychronized给对象上锁（重量级）之后，该对象头的Mark Word中就被设置指向Monitor对象的指针。

Monitor结构如下：

  • 初始时Monitor中Owner为null
  • 当Thread-2执行sychronized(obj)就会将Monitor的所有者Owner置为Thread-2，Monitor中只能有一个Owner
  • 在Thread-2上锁的过程中，如果Thread-3，Thread-4，Thread-5也来执行sychronized(obj)，就会进入EntryList BLOCKED
  • Thread-2执行完同步代码块的内容，然后唤醒EntryList中等待的线程来竞争锁，竞争是非公平的
  • 图中WaitSet中的Thread-0，Thread-1是之前获得过锁的，但条件不满足而进入WAITING状态的线程

注意：

  • synchronized 必须是进入同一个对象的 monitor 才有上述的效果
  • 不加 synchronized 的对象不会关联监视器，不遵从以上规则
  • obj中的Mark Word包含一个指针，指向一个Monitor，而这个Monitor是操作系统提供的对象，在Java中看不到表示

    • locals=3表示局部变量表的大小为3

  • 接下来是具体的字节码指令和操作：

    a. 0: getstatic #2: 获取类的静态字段#2（这里标记为lock引用）的值，并将其压入操作数栈顶。

    b. 3: dup: 复制操作数栈顶的值，并将复制后的值再次压入操作数栈顶。

    c. 4: astore_1: 将操作数栈顶的值（即复制的lock引用）存储到局部变量表的slot 1中。

    d. 5: monitorenter: 进入监视器（即获取锁），将lock对象的MarkWord置为Monitor指针。

    e. 6: getstatic #3: 获取类的静态字段#3（这里标记为i）的值，并将其压入操作数栈顶。

    f. 9: iconst_1: 准备常数1，并将其压入操作数栈顶。

    g. 10: iadd: 从操作数栈顶取出两个值（i和1），将它们相加，并将结果压入操作数栈顶。

    h. 11: putstatic #3: 将操作数栈顶的值（相加后的结果）存储到类的静态字段#3（i）中。

    i. 14: aload_1: 将局部变量表的slot 1（存储的是lock引用）的值加载到操作数栈顶。

    j. 15: monitorexit: 退出监视器（即释放锁），将lock对象的MarkWord重置，并唤醒EntryList中的等待线程。

    k. 16: goto 24: 无条件跳转到字节码指令位置24，即方法的结尾，结束方法执行。

    l. 19: astore_2: 将操作数栈顶的值（异常对象）存储到局部变量表的slot 2中。

    m. 20: aload_1: 将局部变量表的slot 1（存储的是lock引用）的值加载到操作数栈顶。

    n. 21: monitorexit: 退出监视器（即释放锁），将lock对象的MarkWord重置，并唤醒EntryList中的等待线程。

    o. 22: aload_2: 将局部变量表的slot 2（存储的是异常对象）的值加载到操作数栈顶。

    p. 23: athrow: 抛出操作数栈顶的异常对象，结束方法执行。

  • Exception table: 异常处理表，描述了哪些字节码指令范围内可能会抛出异常，并指定异常处理的目标位置。在synchronized代码块中可能会出现异常，如果不进行处理则可能导致锁对象无法被释放。

    a. from to target type: 表示从指令位置from到to范围内可能抛出type类型的异常，并在发生异常时跳转到target指令位置进行处理。

    b. 6 16 19 any: 表示从指令位置6到16范围内可能抛出任何类型的异常，并在发生异常时跳转到指令位置19进行处理。

    c. 19 22 19 any: 表示从指令位置19到22范围内可能抛出任何类型的异常，并在发生异常时跳转到指令位置19进行处理。

  • LineNumberTable: 行号表，记录了字节码指令和Java源代码之间的对应关系。

    a. line 8: 0: 表示Java源代码的第8行对应着字节码指令位置0。

    b. line 9: 6: 表示Java源代码的第9行对应着字节码指令位置6。

    c. line 10: 14: 表示Java源代码的第10行对应着字节码指令位置14。

    d. line 11: 24: 表示Java源代码的第11行对应着字节码指令位置24。

  • LocalVariableTable: 局部变量表，描述了方法中局部变量的信息。

    a. Start Length Slot Name Signature: 表示局部变量的起始指令位置、长度、局部变量表的slot位置、名称和类型签名。

    b. 0 25 0 args [Ljava/lang/String;: 表示局部变量args的作用域从指令位置0开始，长度为25，存储在局部变量表的slot 0位置，名称为args，类型为[Ljava/lang/String;（字符串数组）。

  • StackMapTable: 栈映射表，用于在字节码中恢复方法的局部变量表和操作数栈的类型信息，以便进行字节码验证。

    a. number_of_entries = 2: 表示有两个条目（stack map帧）。

注意！方法级别的 synchronized 不会在字节码指令中有所体现

2. 原理进阶

2.1. 轻量级锁

轻量级锁的使用场景：如果一个对象虽然有多线程要加锁，但加锁的时间是错开的（也就是没有竞争），那么可以使用轻量级锁来优化。

轻量级锁对使用者是透明的，即语法仍然是synchronized

假设有两个方法同步块，利用同一个对象加锁

  • 创建锁记录（Lock Record）对象，每个线程的栈帧都会包含一个锁记录的结构，内部可以存储锁定对象的Mark Word

  • 让锁记录中 Object reference 指向锁对象，并尝试用 cas 替换 Object 的 Mark Word，将 Mark Word 的值存入锁记录

  • 如果 cas 替换成功，对象头中存储了 锁记录地址和状态 00 ，表示由该线程给对象加锁，这时图示如下

  • 如果 cas 失败，有两种情况

    • 如果是其它线程已经持有了该 Object 的轻量级锁，这时表明有竞争，进入锁膨胀过程
    • 如果是自己执行了 synchronized 锁重入，那么再添加一条 Lock Record 作为重入的计数

  • 当退出 synchronized 代码块（解锁时）如果有取值为 null 的锁记录，表示有重入，这时重置锁记录，表示重入计数减一

  • 当退出 synchronized 代码块（解锁时）锁记录的值不为 null，这时使用 cas 将 Mark Word 的值恢复给对象头

    • 成功，则解锁成功
    • 失败，说明轻量级锁进行了锁膨胀或已经升级为重量级锁，进入重量级锁解锁流程
